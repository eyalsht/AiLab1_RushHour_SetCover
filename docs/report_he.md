## 2. חלק א'  פותר Rush Hour

### 2.1 ייצוג הבעיה

הלוח הוא רשת 6×6. כל רכב מתואר על ידי ID, כיוון (אופקי / אנכי), אורך (2 או 3 תאים) ומיקום שמאלי-עליון (שורה, עמודה). ייצגנו את ה-State כ-dataclass קפוא (frozen) שמכיל frozenset של Vehicle dataclasses:

```python
# src/part_a/state.py
@dataclass(frozen=True)
class Vehicle:
    id: str
    is_horizontal: bool
    length: int
    row: int
    col: int

@dataclass(frozen=True)
class State:
    vehicles: FrozenSet[Vehicle]
    grid_size: int = 6

    def get_vehicle(self, vehicle_id: str) -> Optional[Vehicle]:
        for v in self.vehicles:
            if v.id == vehicle_id:
                return v
        return None
```

השימוש ב-`frozenset` הוא קריטי: מכיוון שהוא immutable ואינו תלוי בסדר הרכבים, ניתן לבצע hashing על ה-State ולהכניסו ישירות לתוך `set` של מצבים שכבר ביקרנו בהם  כך בדיקת `state in explored` היא O(1) במקום O(n).

תנאי המטרה: הרכב X הגיע לקצה הימני של הלוח (עמודה 5, כלומר הקצה הימני של X >= 6). כל פעולה היא הזזה של רכב אחד תא אחד לכל כיוון מותר.

### 2.2 אלגוריתמי חיפוש

#### BFS  Breadth-First Search

BFS בודק מצבים לפי עומק  תחילה כל המצבים בעומק 1, אחר-כך עומק 2, וכן הלאה. זה מבטיח שהפתרון הראשון שנמצא הוא הקצר ביותר (במספר מהלכים).

```python
# src/part_a/search.py
frontier = deque([node])   # FIFO  O(1) append ו-popleft
explored = {node.state}    # set לבדיקת O(1)

while frontier:
    current_node = frontier.popleft()
    for action in problem.get_actions(current_node.state):
        child_state = problem.get_result(current_node.state, action)
        if child_state not in explored:
            if problem.is_goal(child_state):
                return child_node
            explored.add(child_state)
            frontier.append(child_node)
```

השימוש ב-`deque` (double-ended queue) חשוב: `popleft()` מ-deque הוא O(1), בעוד `pop(0)` מ-list רגיל הוא O(n) כי הוא מזיז את כל האיברים. ב-BFS ה-frontier יכול להגיע לעשרות אלפי צמתים, אז ההבדל משמעותי.

#### A*  A-Star Search

A* משתמש ב-priority queue (min-heap) שממוין לפי f(n) = g(n) + h(n), כאשר g הוא עלות המסלול עד כה ו-h הוא האומדן ההיוריסטי. כך האלגוריתם מתמקד קודם בנודים שנראים מבטיחים.

```python
# src/part_a/search.py
frontier = []
heapq.heappush(frontier, initial_node)
explored = {initial_node.state: initial_node.path_cost}

while frontier:
    current_node = heapq.heappop(frontier)  # O(log n)
    if problem.is_goal(current_node.state):
        return current_node
    for action in actions:
        child_cost = current_node.path_cost + 1
        if child_state not in explored or child_cost < explored[child_state]:
            explored[child_state] = child_cost
            heapq.heappush(frontier, child_node)
```

ה-`heapq` מממש min-heap בינארי: `heappush` ו-`heappop` הם O(log n). שמירת המפה `state → g_cost` ב-`explored` מאפשרת עדכון מסלולים טובים יותר אם נמצאים בהמשך.

### 2.3 היוריסטיקות

#### h1  ספירת חוסמים

```
h1(s) = |{רכבים אנכיים בנתיב היציאה של X}| + 1
```

ה-+1 הוא בשביל X עצמו. כל רכב שחוסם את X חייב לזוז לפחות פעם אחת, ו-X עצמו חייב לזוז לפחות פעם אחת  אז h1 לעולם לא מעריך ביתר (אדמיסיבילי).

#### h2  חוסמים + חוסמים תקועים

```
h2(s) = h1(s) + |{חוסמים שאינם יכולים לזוז בשני הכיוונים}|
```

חוסם "תקוע" הוא רכב אנכי שגם מעליו וגם מתחתיו חסום (גבול הלוח או רכב אחר). רכב כזה דורש לפחות עוד מהלך אחד כדי לפנות לו מקום לפני שהוא יכול לזוז. לכן הוספת 1 לכל חוסם תקוע היא עדיין חסם תחתון תקף.

```python
# src/part_a/problem.py
h2 = h1
for car_id in blocking_cars:
    car = state.get_vehicle(car_id)
    if not car.is_horizontal:
        can_move_up   = car.row > 0 and grid[car.row - 1][car.col] is None
        can_move_down = (car.row + car.length < state.grid_size
                         and grid[car.row + car.length][car.col] is None)
        if not can_move_up and not can_move_down:
            h2 += 1
```

**אדמיסיביליות:** שתי ההיוריסטיקות סופרות רק מהלכים שחייבים לקרות  אף פעם לא מעריכות ביתר.

**עקביות (מונוטוניות):** כל פעולה עולה 1. בכל צעד, h יכול לרדת לכל היותר ב-1 (הסרנו חוסם אחד או קרבנו את X). לכן h(n) ≤ 1 + h(n'), כלומר f = g + h אינו יורד לאורך כל מסלול. A* עם היוריסטיקה עקבית לא יצטרך להרחיב מחדש נודים שכבר נסגרו.

### 2.4 יעילות מבני הנתונים

| מבנה | פעולה | סיבוכיות | שימוש |
|---|---|---|---|
| `deque` | `append` / `popleft` | O(1) | תור BFS |
| `heapq` | `heappush` / `heappop` | O(log n) | תור עדיפויות של A* |
| `set` / `dict` | `in`, `add` | O(1) ממוצע | מצבים שבוקרו |

ה-`frozenset` בתוך State מאפשר hashing של כל מצב ב-O(|vehicles|). מאחר שיש לכל היותר ~15 רכבים בלוח, זה בפועל O(1).

### 3.2 קידוד הכרומוזום

כל כרומוזום הוא מערך בינארי באורך n. ג'ן i = 1 אומר שקבוצה i נבחרת; 0 = לא נבחרת.

```python
# src/part_b/chromosome.py
@dataclass
class Chromosome:
    genes: List[int]          # בינארי, אורך n
    fitness: float = float('inf')
    is_valid: bool = False
    cost: float = float('inf')
```

### 3.3 פיטנס  תיקון ו-Trim

כרומוזום אקראי כמעט תמיד יחמיץ אלמנטים מסוימים. במקום לענוש פתרונות לא חוקיים (penalty), בחרנו לתקן אותם בתוך `evaluate()`. התוצאה: האוכלוסייה תמיד מורכבת מפתרונות חוקיים, ו-fitness הוא פשוט העלות הכוללת.

**שלב תיקון:** מזהים אלמנטים לא מכוסים. מוסיפים בצורה גרידית את הקבוצה הלא-נבחרת עם יחס עלות/כיסוי-חדש הטוב ביותר, עד שהכל מכוסה.

**שלב Trim:** עוברים על הקבוצות הנבחרות לפי סדר עלות יורד. מסירים קבוצה אם כל האלמנטים שלה מכוסים גם על ידי קבוצות אחרות (קבוצה מיותרת).

```python
# src/part_b/chromosome.py  evaluate()

# שלב 1: תיקון  הוסף קבוצות עד שהכל מכוסה
uncovered = [e for e, count in enumerate(covered_counts) if count == 0]
while uncovered:
    best_set = -1
    best_ratio = float('inf')
    for s_idx in range(problem.n):
        if self.genes[s_idx] == 1:
            continue
        new_covered = sum(1 for e in problem.sets_to_elements[s_idx]
                          if covered_counts[e] == 0)
        if new_covered > 0:
            ratio = problem.costs[s_idx] / new_covered
            if ratio < best_ratio:
                best_ratio = ratio
                best_set = s_idx
    if best_set != -1:
        self.genes[best_set] = 1
        for e in problem.sets_to_elements[best_set]:
            covered_counts[e] += 1
        uncovered = [e for e in uncovered if covered_counts[e] == 0]

# שלב 2: Trim  הסר קבוצות מיותרות (היקרות קודם)
selected_sets.sort(key=lambda idx: problem.costs[idx], reverse=True)
for s_idx in selected_sets:
    is_redundant = all(covered_counts[e] >= 2
                       for e in problem.sets_to_elements[s_idx])
    if is_redundant:
        self.genes[s_idx] = 0
        for e in problem.sets_to_elements[s_idx]:
            covered_counts[e] -= 1

# שלב 3: חישוב עלות סופית
self.cost = sum(problem.costs[i] for i, v in enumerate(self.genes) if v == 1)
self.is_valid = True
self.fitness = self.cost   # תמיד חוקי, אין עונש
```

### 3.4 אתחול אוכלוסייה

אוכלוסייה של 300 כרומוזומים. כל ג'ן מוגרל ל-1 בהסתברות 2/n (ספארסי מאוד). שלב התיקון ממלא כל כרומוזום לפתרון חוקי. התחלה ספארסית מאלצת את מנגנון התיקון לבחור בחירות ראשוניות מגוונות, מה שמייצר אוכלוסייה ראשונית סבירה.

### 3.5 סלקציה

**Tournament Selection** עם k=3 ממאגר הבחירה: רק ה-50% הטובים ביותר של האוכלוסייה (ממוינת לפי fitness). מושכים 3 מתמודדים אקראיים מהמאגר  המנצח הוא בעל ה-fitness הנמוך ביותר. הגבלה ל-50% מספקת לחץ סלקציה נוסף מבלי להוציא לגמרי חצי מהאוכלוסייה.

```python
# src/part_b/ga_engine.py
mating_pool = self.population[:mating_pool_size]  # top 50%
contenders = self.random.sample(mating_pool, k=3)
return min(contenders, key=lambda c: c.fitness)
```

### 3.6 הכלאה

**Uniform Crossover** בשיעור 0.8. לכל מיקום ג'ן, מגרילים (50/50) מאיזה הורה לרשת. זה יוצר ילדים שהם פסיפס של שני ההורים, ולא חתך בנקודה אחת.

```python
# src/part_b/ga_engine.py
def crossover(self, p1, p2):
    if self.random.random() < self.crossover_rate:
        c1_genes, c2_genes = [], []
        for g1, g2 in zip(p1.genes, p2.genes):
            if self.random.random() < 0.5:
                c1_genes.append(g1); c2_genes.append(g2)
            else:
                c1_genes.append(g2); c2_genes.append(g1)
        return Chromosome(c1_genes), Chromosome(c2_genes)
    return Chromosome(list(p1.genes)), Chromosome(list(p2.genes))
```

### 3.7 מוטציה

**Bit-flip** בשיעור 0.05 לג'ן. כל ג'ן מתהפך ב-5% הסתברות באופן עצמאי. מכיוון שמנגנון התיקון מתקן תוצאות לא חוקיות, מוטציה יכולה בחופשיות להוריד קבוצות  אם היו נחוצות, הן יתווספו חזרה בשלב התיקון.

### 3.8 אליטיזם

10% מהאוכלוסייה הטובים ביותר עוברים לדור הבא ללא שינוי. מבטיח שפתרונות טובים לא אובדים בגלל הכלאה/מוטציה.

### 3.9 עצירה מוקדמת

שני תנאים עוצרים את הריצה לפני 200 דורות:

1. **Stagnation:** ה-fitness הטוב ביותר לא השתפר במשך 30 דורות רצופים
2. **Entropy convergence:** האנטרופיה הממוצעת של האוכלוסייה ירדה מתחת ל-0.05

### 3.10 מדדי גיוון

מדדנו שלושה מדדי גיוון בכל דור:

```python
# src/part_b/ga_engine.py  record_metrics()

matrix = np.array([c.genes for c in self.population])  # (pop_size, n)
p_1 = matrix.mean(axis=0)   # תדירות ה-1 בכל מיקום
p_0 = 1 - p_1

# אנטרופיה שאנונית  ממוצע על פני כל המיקומים
p_1_clipped = np.clip(p_1, 1e-9, 1 - 1e-9)
p_0_clipped = np.clip(p_0, 1e-9, 1 - 1e-9)
entropy = float(
    -np.sum(p_1_clipped * np.log2(p_1_clipped)
            + p_0_clipped * np.log2(p_0_clipped)) / self.problem.n
)

# מרחק Hamming ממוצע בין-זוגי צפוי
hamming_distance = float(np.sum(2 * p_1 * p_0))

# לחץ סלקציה: avg_fitness / best_fitness (גבוה = לחץ גבוה יותר)
selection_pressure = float(avg / best) if best > 0 else 1.0
```

- **אנטרופיה שאנונית:** קרובה ל-1 = אוכלוסייה מגוונת; קרובה ל-0 = האוכלוסייה התכנסה
- **מרחק Hamming ממוצע:** כמה מיקומים שונים בממוצע בין שני כרומוזומים אקראיים
- **לחץ סלקציה:** יחס avg/best  ככל שגבוה יותר כך הסלקציה אגרסיבית יותר

### 3.14 תצפיות לגבי התכנסות האוכלוסייה

**תצפית מרכזית:** בהרצת האלגוריתם עם הפרמטרים הרגילים (entropy_threshold=0.05), שמנו לב שכל הריצות עוצרות כבר בדור 1 עקב entropy convergence. האנטרופיה ירדה ל-~0.04 כבר אחרי האתחול, לפני שהאלגוריתם הספיק לבצע אפילו סיבוב הכלאה אחד.

הסיבה: מנגנון התיקון אגרסיבי מאוד. לאחר שמחילים repair+trim על כל 300 הכרומוזומים הספארסיים, הם מתכנסים לפתרונות דומים מאוד (אותן קבוצות "טובות" נבחרות בגרידי על ידי כולם). כתוצאה מכך, האנטרופיה  שמודדת כמה "מעורב" הג'ן בכל מיקום  קורסת לפני שאפילו דור אחד של אבולוציה התרחש.

כדי לייצר גרפים משמעותיים שמראים אבולוציה אמיתית, הרצנו גרסה עם entropy_threshold=0.0 (ה-early stopping על אנטרופיה מושבת). בתנאים אלה, האלגוריתם ממשיך עד לעמידה בתנאי ה-stagnation (30 דורות ללא שיפור).

**מה ניתן ללמוד מזה:** ה-GA שלנו הוא בפועל **חיפוש גרידי מקבילי**  המיטוב האמיתי קורה בשלב האתחול+תיקון, לא בשלב האבולוציה. כדי לאפשר אבולוציה משמעותית, יש לשקול: (א) תיקון חלקי בלבד (לא עד לפתרון מלא), (ב) הוספת randomization לשלב התיקון, (ג) הגדלת גיוון האתחול. עם זאת, הביצועים הסופיים עדיין טובים בהרבה מהגרידי הבסיסי.

## 4. מסקנות

**חלק א':** H1 פשוטה אך יעילה  הורידה את ממוצע הנודים מ-2,478 (BFS) ל-2,074 ו-EBF מ-1.635 ל-1.399. H2 שיפרה מעט יותר מבחינת נודים (1,975, EBF 1.388) אבל לקחה יותר זמן (547ms מול 282ms) בגלל עלות גבוהה יותר לנוד. עבור המערכת הנוכחית, A* עם h1 הוא הבחירה הפרקטית. כל שלוש השיטות מצאו פתרונות אופטימליים  אוששנו זאת ב-13 חידות שלהן יש פתרונות ייחוס מלאים ב-rh.txt.

**חלק ב':** מנגנון התיקון הוכיח את עצמו  האוכלוסייה כולה תמיד חוקית ואין צורך לכוונן מקדמי עונש. ה-GA הגיע לתוצאות בפער של 1-8% מהאופטימום, לעומת 8-16% של הגרידי. הגילוי העיקרי היה שמנגנון התיקון האגרסיבי למעשה ממזג את האוכלוסייה כבר בדור 0, ומותיר מעט מקום לאבולוציה אמיתית. שיפור עתידי: תיקון חלקי שישמר יותר גיוון גנטי לאורך הדורות.
